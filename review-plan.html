Calendar, tasks, and phone sync

Recommended architecture: use CalDAV as the durable sync layer. Pick one provider (existing Google account, Nextcloud, mailbox.org, Fastmail, etc.), sync desktop data with vdirsyncer, view/edit with khal or a GUI, and use the phone’s native calendar or DAVx⁵ on Android. Keep notes in Obsidian; do not force Obsidian to become the calendar database.

Calendar

Enable khal beside the already-installed vdirsyncer. Declare calendars, colors, and a user timer for periodic sync. Add ikhal for a quick terminal UI.

Tasks

Choose after a 7-day trial: Tasks.org + DAVx⁵ for open CalDAV/VTODO, or Todoist for the smoother cross-platform product. Avoid maintaining two inboxes.

Desktop glance

Show the next event and overdue count in Waybar; clicking opens the chosen client. Notifications should come from one source only to avoid doubles.

Secrets

Do not put account passwords in Nix or Git. Add sops-nix or agenix, or read credentials from the keyring/password command.

Decision rule: if shared calendars and mobile polish matter most, use your current cloud provider. If ownership matters most, use Nextcloud. CalDAV keeps the desktop configuration substantially the same either way.

Implementation sequence

  1. Snapshot: commit current intended changes, record nh os info, disk usage, enabled units, and installed closure size.
  2. Fix correctness: MIME IDs, locale override, duplicate packages/services, and the two shutdown implementations.
  3. Reduce risk: sudo/groups/open ports, then validate development, gaming, media and hardware workflows.
  4. Add personal sync: secrets mechanism → vdirsyncer account → khal/UI → phone → Waybar. Test create/edit/delete in both directions.
  5. Modularize: add feature options and move heavy stacks behind them or into dev shells.
  6. Observe for 30 days: keep a “candidate removal” list; uninstall apps with no observed use and an easy replacement path.
  7. Automate quality: one check command, CI evaluation, update cadence, and rollback notes.
Safe rollout: make one thematic commit per phase. Run nix flake check, build first, then follow this repository’s instruction and run nh os switch. Keep the previous generation available until the relevant workflow has been tested.

Definition of “clean”

  • Every always-installed package has a known recurring job.
  • Heavy hobbies/toolchains can be toggled without editing package lists.
  • Calendar and tasks round-trip between desktop and phone, with one notification source.
  • No plaintext secrets are stored in the repository or Nix store.
  • No unrestricted passwordless sudo; privileged groups and firewall openings are justified.
  • nix flake check, formatting, dead-code, and static analysis pass before switching.
  • The README can recover the machine and explain every non-obvious choice.
